> **Date:** 2026-03-05
> **Branch:** feature/sport-module-refactor (to be created)
> **Status:** Approved

---

## Overview

Refactor the scorer and shared configuration into a sport-agnostic architecture that supports many simultaneous pools with arbitrary combinations of sport, scoring rules, and payout structures. Implement the World Cup 2026 scorer module as the first new sport beyond March Madness.

**Goals:**
- Any pool can have any sport, any payout structure — fully data-driven from chain
- Adding a new sport = one new directory in `shared/sports/`, zero other changes
- No naming convention hacks, no CLI args to remember, no duplicated config

## Contract Changes

Two new fields on `BracketPool`:

```solidity
string public sportId;        // "mm", "wc", "nba", etc.
uint16[] public payoutBps;    // e.g. [6000, 2500, 1500] — must sum to 10000
```

Constructor additions:

```solidity
constructor(
  // ...existing params...
  string memory _sportId,
  uint16[] memory _payoutBps
) {
  require(_sumBps(_payoutBps) == 10000, "Payouts must sum to 10000");
  sportId = _sportId;
  payoutBps = _payoutBps;
}

function getPayoutBps() external view returns (uint16[] memory) {
  return payoutBps;
}
```

`BracketPoolFactory.createPool()` gets the same two new params and passes them through. No other contract logic changes.

**Impact on existing code:**
- All 82 Solidity tests need `sportId` + `payoutBps` added to constructor calls
- Deploy scripts updated
- Sepolia redeployment required after merge
- Frontend `usePoolDetails` hook adds `sportId` and `payoutBps` to batch read

## Scorer Pipeline Changes

```typescript

const SPORT_MODULES: Record<string, SportModule> = {
  mm: marchmadness,
  wc: worldcup2026,
};

export async function runScorer(poolAddress, rpcUrl, actualTiebreaker) {
  const { sportId, payoutBps } = await readPoolConfig(poolAddress, rpcUrl);

  const module = SPORT_MODULES[sportId];
  if (!module) throw new Error(`Unknown sportId: ${sportId}`);

  const entries = await readEntries(poolAddress, rpcUrl);
  const results = await readGameResults(poolAddress, rpcUrl);

  const scored = entries.map(e => ({
    ...e,
    score: module.scorePicks(e.picks, results),
  }));

  const ranked = rankEntries(scored, actualTiebreaker);
  const distributed = distributePrizes(ranked, prizePool, payoutBps);
  // ...rest unchanged
}
```

- `distributePrizes()` accepts `payoutBps: number[]` param instead of hardcoded 60/25/15

## World Cup 2026 Module

### Teams

48 teams across 12 groups (A–L), 4 teams per group. Team IDs: `keccak256(abi.encodePacked(fifaCode))`.

```typescript
// shared/sports/worldcup2026/teams.ts
export interface WCTeam {
  id: `0x${string}`;
  name: string;
  code: string;   // FIFA 3-letter code e.g. "BRA"
  group: string;  // "A" through "L"
}
```

### Pick Encoding — `bytes32[88]`

| Index | Count | Content |
|-------|-------|---------|
| 0–47 | 48 | Group finishing order (4 teams × 12 groups, positions 1st→4th) |
| 48–55 | 8 | Which 8 third-place teams advance |
| 56–71 | 16 | Round of 32 picks |
| 72–79 | 8 | Quarterfinal picks |
| 80–83 | 4 | Semifinal picks |
| 84–85 | 2 | Finalists |
| 86 | 1 | Winner |
| 87 | 1 | 3rd place match winner |

### Scoring

| Pick | Points |
|------|--------|
| Correct 1st in group | 10 |
| Correct 2nd in group | 8 |
| Correct 3rd in group | 5 |
| Correct 4th in group | 3 |
| Correct advancing 3rd-place team | 5 |
| Team in Round of 32 | 10 |
| Team in Quarterfinals | 20 |
| Team in Semifinals | 40 |
| Finalist | 80 |
| Winner | 100 |
| 3rd place winner | 40 |

**Perfect score = 1,132 points.** Tiebreaker = predicted total goals in the Final.

### Validation

`validatePicks()` checks internal consistency before scoring:
- R32 teams ⊆ predicted group advancers (1st + 2nd per group) + advancing 3rd-place picks
- QF teams ⊆ R32 picks
- SF teams ⊆ QF picks
- Finalists ⊆ SF picks
- Winner must be one of the 2 finalists
- 3rd place winner must be one of the 2 SF losers

Invalid slots score 0 — not disqualified entirely.

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Pool config location | On-chain (`sportId` + `payoutBps`) | Source of truth for financial config; scorer reads chain, no CLI args or naming hacks |
| Sport module location | `shared/` at repo root | Imported by both scorer and frontend; prebuild copies to `web/src/shared/` |
| Payout structure | `uint16[] payoutBps` (sum = 10000) | Arbitrary tiers per pool; winner-take-all = `[10000]`, top-3 = `[6000, 2500, 1500]` |
| Adding new sport | One new dir in `shared/sports/` + entry in `SPORT_MODULES` | Zero changes to contracts, pipeline, or tests |
| WC pick format | Flat `bytes32[88]` | Contract stays sport-agnostic; scorer interprets positions |
| WC tiebreaker | Total goals in the Final | Mirrors MM approach; simple and unambiguous |
